## 1.1 本轮具体实施步骤（如何优化）

### O-1 汇编上下文切换替代 ucontext
- 目标：移除 `swapcontext/makecontext/getcontext`，改为最小寄存器保存/恢复路径。
- 操作步骤：
  1. 新增跨架构上下文结构定义 `FiberContext`，分别描述 x86_64 与 aarch64 的寄存器布局。
  2. 新增汇编函数 `mycoroutine_context_swap(from, to)`：
     - 保存当前协程 callee-saved 寄存器与栈指针到 `from`。
     - 从 `to` 恢复寄存器与栈指针并跳转到保存的 PC/RIP。
  3. 在 `Fiber` 初始化阶段自行构造协程入口栈帧：
     - 栈顶对齐；
     - 写入入口地址（`Fiber::MainFunc` 包装入口）；
     - 初始化 `FiberContext`。
  4. 将 `Fiber::resume/yield/call/back` 的切换点全部替换为 `mycoroutine_context_swap`。
  5. 保留共享栈快照机制，仅将“栈指针读取来源”从 `ucontext` 改为 `FiberContext`。
  6. 在 CMake 中启用 ASM，并按架构编译对应 `.S` 文件。
- 涉及文件：
  - `include/mycoroutine/context.h`
  - `src/context_switch_x86_64.S`
  - `src/context_switch_aarch64.S`
  - `include/mycoroutine/fiber.h`
  - `src/fiber.cpp`
  - `CMakeLists.txt`

### O-6 协程同步原语（CoMutex / Channel / WaitGroup）
- 目标：协程等待时挂起协程本身，不阻塞工作线程。
- 操作步骤：
  1. 设计统一等待节点 `FiberWaiter`，保存：
     - 当前协程对象；
     - 当前调度器；
     - `notified` 标记。
  2. 实现通用等待/唤醒流程：
     - 等待：把 `FiberWaiter` 放入等待队列后 `yield`；
     - 唤醒：设置 `notified=true`，并通过 `scheduler->scheduleLock(fiber)` 重新调度。
  3. `CoMutex`：
     - `lock()` 竞争失败时进入等待队列；
     - `unlock()` 唤醒一个等待协程并转移所有权。
  4. `WaitGroup`：
     - `add/done` 修改计数；
     - 计数归零时批量唤醒所有 `wait()` 挂起协程。
  5. `Channel<T>`：
     - 支持有界缓冲；
     - `capacity=0` 作为同步通道；
     - `close()` 后统一唤醒发送/接收等待协程并返回关闭状态。
  6. 新增独立测试覆盖互斥、生产消费、无缓冲通道、等待组场景。
- 涉及文件：
  - `include/mycoroutine/sync.h`
  - `tests/sync/main.cpp`
  - `tests/CMakeLists.txt`

### O-10 任务队列数据结构优化
- 目标：降低任务队列中间删除与对象拷贝开销。
- 操作步骤：
  1. 将调度器任务容器由 `std::vector<ScheduleTask>` 改为 `std::deque<ScheduleTask>`。
  2. 入队路径改为 `emplace_back(std::move(task))`，减少临时对象拷贝。
  3. 出队路径改为 `out_task = std::move(*it)`，再 `erase(it)`，降低复制成本。
  4. FIFO 策略保留“按线程匹配首个可执行任务”的线性扫描逻辑，但容器搬移成本更低。
  5. MLFQ 重新入队路径同步改为 move 语义。
- 涉及文件：
  - `include/mycoroutine/scheduler.h`
  - `src/scheduler.cpp`

## 2. 剩余优化方向（后续迭代）

### 2.1 无锁任务队列（P2）
- 现状：`m_tasks` 仍由全局锁保护
- 方向：MPMC 无锁队列，或 per-thread 本地队列 + work-stealing
- 价值：多线程扩展性（高并发下吞吐提升）

### 2.2 栈内存池化 + guard page（P0/P1）
- 现状：协程栈仍为 `malloc/free`
- 方向：`mmap` + `mprotect(PROT_NONE)` guard page + per-thread 栈池
- 价值：栈溢出检测 + 降低分配成本

### 2.3 时间轮替代 `std::set`（P2）
- 现状：定时器插入/删除为 O(log n)
- 方向：分层时间轮（Hierarchical Timing Wheel）
- 价值：海量定时器场景下提升显著

### 2.4 io_uring 后端（P3）
- 现状：IO 后端为 epoll
- 方向：提供 io_uring 后端，支持批量提交/完成
- 价值：减少系统调用开销，提升高并发 IO 吞吐

### 2.5 协程 Future/Promise（P2）
- 现状：无标准异步结果传递抽象
- 方向：`CoFuture<T>` / `CoPromise<T>` 与调度器集成
- 价值：提升异步编程表达力

### 2.6 内建指标与统计（P2）
- 现状：缺少结构化指标
- 方向：导出切换次数/延迟、队列深度、协程池命中率、IO/定时器指标
- 价值：可观测性增强，便于性能分析和压测回归

### 2.7 协程栈水位追踪（P3）
- 现状：固定栈，无水位统计
- 方向：canary + 最大栈占用记录，可选 segmented/split stack
- 价值：内存安全 + 栈尺寸调优依据

### 2.8 减少 shared_ptr 开销（P2）
- 现状：调度热路径含较多 `shared_ptr` 原子计数操作
- 方向：内部使用 intrusive_ptr/裸指针，外部接口继续暴露 shared_ptr
- 价值：降低调度路径原子开销

### 2.9 编译期调度策略（P3）
- 现状：策略分发为运行时 switch
- 方向：策略模板化，编译期多态
- 价值：减少分支开销，提升热路径效率
