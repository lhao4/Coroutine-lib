## 3. 问题总览
1. 共享栈快照需要精确获取 SP，且要兼容多架构寄存器布局。
2. 多协程复用同一共享栈槽位时，owner 切换容易导致栈数据污染。
3. 共享栈协程跨线程执行会破坏上下文与槽位归属。
4. 共享栈 + 协程复用（`reset`）路径下，context 重新初始化时机容易出错。
5. 协程嵌套下 `yield` 返回路径必须优先回父协程，不然会回错到调度协程。
6. “父子协程都用共享栈”属于未完成语义，最初仅靠 `assert`，Release 下会失效。
7. 调度器从 FIFO 扩展到多策略时，需要保持旧接口行为不变。
8. MLFQ 参数存在非法输入，需要归一化，否则会产生除零/无效层级问题。
9. 协程池复用需要解决 key 设计和复用污染问题。
10. 测试覆盖最初不足，新增能力需要可回归的自动化测试矩阵。
11. 文档存在”历史状态残留”，与当前实现不一致，需要系统性修正。
12. `Scheduler::stop()` 写 `m_stopping` 未持锁，与工作线程的加锁读形成数据竞争（TSan 检出）。
13. stress test 的 sentinel 回调使用 `shared_ptr<function>` 自捕获模式导致循环引用内存泄漏（ASan/LeakSanitizer 检出）。
14. `iomanager.cpp` 中 `epoll_ctl`/`write` 返回值仅在 `assert` 中使用，Release 编译产生 `-Wunused-variable` 警告。
15. TSan 在 WSL2 高 ASLR 内核下无法启动，需要调整 `vm.mmap_rnd_bits`。

## 4. 关键问题与解决过程

### 4.1 共享栈 SP 获取与快照边界

#### 4.1.1 问题
共享栈切换要保存“活动栈区间”，前提是拿到正确 SP。不同 CPU 架构在 `ucontext_t` 中 SP 字段不同，直接写死会导致数据错误或崩溃。

#### 4.1.2 思考
- 必须用条件编译按架构读取 SP。
- 不支持的架构直接编译时报错，避免静默错误。

#### 4.1.3 解决
关键实现如下：

```cpp
static uintptr_t GetContextStackPointer(const ucontext_t& ctx) {
#if defined(__x86_64__) && defined(REG_RSP)
    return static_cast<uintptr_t>(ctx.uc_mcontext.gregs[REG_RSP]);
#elif defined(__i386__) && defined(REG_ESP)
    return static_cast<uintptr_t>(ctx.uc_mcontext.gregs[REG_ESP]);
#elif defined(__aarch64__)
    return static_cast<uintptr_t>(ctx.uc_mcontext.sp);
#else
#error "shared stack currently supports x86_64/i386/aarch64"
#endif
}
```

代码位置：`src/fiber.cpp:64-74`

### 4.2 共享栈 owner 切换与数据污染

#### 4.2.1 问题
多个共享栈协程竞争同一槽位时，如果新 owner 直接覆盖栈，会把旧 owner 的活动栈帧破坏。

#### 4.2.2 思考
owner 切换必须分两步：
- 旧 owner 先保存快照。
- 新 owner 再恢复自己的快照并接管 owner。

#### 4.2.3 解决
在 `prepareSharedStack()` 中显式处理 owner 切换：

```cpp
std::shared_ptr<Fiber> owner = slot->owner.lock();
if (owner && owner.get() != this) {
    if (owner.get() == t_fiber) {
        owner->saveCurrentSharedStackSnapshot();
    } else {
        owner->saveSharedStackSnapshot();
    }
}

if (!owner || owner.get() != this) {
    restoreSharedStackSnapshot();
    slot->owner = shared_from_this();
}
```

代码位置：`src/fiber.cpp:300-312`

### 4.3 共享栈跨线程执行约束

#### 4.3.1 问题
共享栈槽位是线程局部池管理，协程如果跨线程执行会破坏 owner 约束与 context 假设。

#### 4.3.2 思考
共享栈协程需要“首次运行线程绑定”，后续必须在同线程执行。

#### 4.3.3 解决
在 `prepareSharedStack()` 增加线程绑定校验：

```cpp
const pid_t tid = Thread::GetThreadId();
if (m_ownerThread == -1) {
    m_ownerThread = tid;
} else {
    assert(m_ownerThread == tid);
}
```

代码位置：`src/fiber.cpp:273-279`

### 4.4 共享栈 + reset 复用路径

#### 4.4.1 问题
`Fiber` 被复用后，可能仍持有旧共享槽位或旧 context 状态。若 `reset` 不重建 context，会出现切换异常。

#### 4.4.2 思考
- 独立栈：直接重建 context。
- 共享栈：若已有槽位，绑定槽位后重建；若无槽位，延迟到 `prepareSharedStack()` 初始化。

#### 4.4.3 解决
`reset()` 分支处理如下：

```cpp
if (!m_useSharedStack) {
    initFiberContext();
    return;
}

SharedStackSlot* slot = ToSharedStackSlot(m_sharedStackSlot);
if (slot) {
    m_stack = slot->stack;
    m_stacksize = static_cast<uint32_t>(slot->size);
    initFiberContext();
} else {
    m_ctxInitialized = false;
}
```

代码位置：`src/fiber.cpp:409-422`

### 4.5 协程嵌套返回链路

#### 4.5.1 问题
嵌套场景中子协程 `yield()` 如果仍走“回调度协程/主协程”旧路径，会导致返回链路错误。

#### 4.5.2 思考
嵌套调用必须把“返回父协程”作为更高优先级分支。

#### 4.5.3 解决
`yield()` 优先检测父协程路径：

```cpp
if (m_returnToParent && m_parent != nullptr) {
    back();
    return;
}
```

父子关系建立和清理在 `call()` 中完成，位置：`src/fiber.cpp:491-504`

### 4.6 assert 在 Release 失效

#### 4.6.1 问题
最初“父子协程都使用共享栈”的未支持场景仅有 `assert`。Release 编译会移除断言，可能落入未定义行为。

#### 4.6.2 思考
底层高性能库不应依赖异常，且 Release 需要可生效保护。

#### 4.6.3 解决
将 `call()` 改为错误码 API，保留运行时硬保护：

```cpp
if (m_state != READY) {
    return CALL_ERR_NOT_READY;
}
...
if (m_useSharedStack && parent->m_useSharedStack) {
    return CALL_ERR_SHARED_NESTED_UNSUPPORTED;
}
```

代码位置：`src/fiber.cpp:468-485`

错误码定义：

```cpp
enum CallCode {
    CALL_OK = 0,
    CALL_ERR_NOT_READY = 1,
    CALL_ERR_NO_CURRENT_FIBER = 2,
    CALL_ERR_SELF_CALL = 3,
    CALL_ERR_SHARED_NESTED_UNSUPPORTED = 4
};
```

### 4.7 调度策略扩展与兼容性

#### 4.7.1 问题
新增 PRIORITY/MLFQ/EDF/HYBRID 时，不能破坏既有 `scheduleLock()` 用法。

#### 4.7.2 思考
- 新增 `scheduleEx(options)` 作为增强入口。
- `scheduleLock()` 保持语义，内部转发。

#### 4.7.3 解决
`scheduleLock()` 转发到 `scheduleEx()`：

```cpp
template <class FiberOrCb>
void scheduleLock(FiberOrCb fc, int thread = -1) {
    ScheduleOptions options;
    options.thread = thread;
    scheduleEx(fc, options);
}
```

### 4.9 协程池 key 设计与复用污染

#### 4.9.1 问题
复用池如果不按关键参数分桶，会把不同运行模式协程混用（如 shared/non-shared）。

#### 4.9.2 思考
key 必须包含：
- 栈大小（默认值归一化）
- `run_in_scheduler`
- `use_shared_stack`

#### 4.9.3 解决
`makeKey()` 按位编码：

```cpp
const uint64_t normalized = static_cast<uint64_t>(normalizeStackSize(stacksize)) & ((1ull << 60) - 1);
const uint64_t run_flag = run_in_scheduler ? (1ull << 62) : 0;
const uint64_t shared_flag = use_shared_stack ? (1ull << 63) : 0;
return normalized | run_flag | shared_flag;
```

并在 `release()` 严格要求 `TERM`：`src/coroutine_pool.cpp:70-73`

### 4.12 Scheduler::stop() 数据竞争（TSan 检出）

#### 4.12.1 问题
TSan 在 stress_test 中报告 `m_stopping` 存在 data race：主线程在 `stop()` 中写 `m_stopping = true`（无锁），工作线程在 `stopping()` 中加锁读 `m_stopping`。写操作未持锁，读写不在同一同步域。

#### 4.12.2 思考
- `stopping()` 已通过 `m_mutex` 保护读取，写端也必须用相同的锁。
- 不选用 `std::atomic<bool>` 方案，因为 `stopping()` 内部还需要在同一临界区内原子检查 `m_tasks.empty()` 和 `m_activeThreadCount`，单独原子化 `m_stopping` 无法保证三者的复合判断一致性。

#### 4.12.3 解决
将 `m_stopping = true` 包裹在 `std::lock_guard<std::mutex>` 中：

```cpp
// 标记调度器为停止状态
{
    std::lock_guard<std::mutex> lock(m_mutex);
    m_stopping = true;
}
```

代码位置：`src/scheduler.cpp:435-438`

### 4.13 stress test sentinel 循环引用内存泄漏（ASan 检出）

#### 4.13.1 问题
ASan/LeakSanitizer 报告 stress test 中 312 bytes 泄漏（6 次分配）。泄漏源自三个测试函数中的 sentinel 模式：

```cpp
auto sentinel = std::make_shared<std::function<void()>>();
*sentinel = [&, sentinel]() {  // sentinel 被值捕获 → 循环引用
    if (...) { /* done */ }
    else { iom.scheduleLock(std::function<void()>(*sentinel)); }
};
```

`sentinel`（`shared_ptr`）→ `*sentinel`（`function`）→ lambda → `sentinel` 副本（`shared_ptr`）→ 同一对象。引用计数永远 ≥ 1，析构时无法释放。

#### 4.13.2 思考
- 方案 A：改用 `weak_ptr` 捕获 — 需要在 lambda 内部 `lock()`，增加运行时开销和代码复杂度。
- 方案 B：任务完成后手动断开循环 — 最简单，只需一行 `*sentinel = nullptr`。
- 选择方案 B，因为 `cv.wait()` 返回后 sentinel 已不再需要。

#### 4.13.3 解决
在 `cv.wait()` 返回后断开循环引用：

```cpp
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, [&]() { return ready; });
lock.unlock();
*sentinel = nullptr;  // break circular reference
```

三个测试函数（`stress_concurrent`、`stress_external_threads`、`stress_callback_pool`）均做同样修复。

代码位置：`tests/stress/main.cpp:63-64`、`tests/stress/main.cpp:124-125`、`tests/stress/main.cpp:172-173`

### 4.14 Release 编译警告：assert-only 变量

#### 4.14.1 问题
`iomanager.cpp` 中两处局部变量 `rt` 仅在 `assert()` 中使用。Release 模式定义 `NDEBUG`，`assert` 被编译为空操作，导致 `-Wunused-variable` 警告：

```
warning: unused variable 'rt' [-Wunused-variable]
  115 |     int rt = epoll_ctl(m_epfd, EPOLL_CTL_ADD, m_tickleFds, &event);
  432 |     int rt = write(m_tickleFds, &one, sizeof(one));
```

#### 4.14.2 思考
- 方案 A：`[[maybe_unused]]` 属性 — C++17 标准，语义清晰，但对”assert-only”场景有些过度宽松（掩盖真正未使用）。
- 方案 B：`(void)rt` — 惯用手法，明确表达”已知未使用”，且在 assert 保留时 `rt` 仍有两处引用，不会被误认为冗余。
- 选择方案 B。

#### 4.14.3 解决
在 `assert` 之后紧跟 `(void)rt`：

```cpp
int rt = epoll_ctl(m_epfd, EPOLL_CTL_ADD, m_tickleFds, &event);
assert(!rt);
(void)rt;
```

```cpp
int rt = write(m_tickleFds, &one, sizeof(one));
assert(rt == sizeof(one));
(void)rt;
```

代码位置：`src/iomanager.cpp:115-117`、`src/iomanager.cpp:432-434`

### 4.15 TSan 在 WSL2 高 ASLR 内核下无法运行

#### 4.15.1 问题
所有 TSan 测试均在启动时崩溃：

```
FATAL: ThreadSanitizer: unexpected memory mapping 0x711438a72000-0x711438f00000
```

#### 4.15.2 思考
- 这是 TSan 运行时与内核地址空间布局的兼容性问题，非代码缺陷。
- WSL2 内核 6.6 的 `vm.mmap_rnd_bits` 默认值过高（通常 32 位），TSan 运行时预期 ≤ 28 位。
- 该问题不影响 Debug/Release/ASan 构建和测试。

#### 4.15.3 解决
以 root 权限降低 ASLR 随机化位数即可：

```bash
sudo sysctl vm.mmap_rnd_bits=28
```

该设置仅在当前会话生效，重启后恢复默认值。如需持久化可写入 `/etc/sysctl.conf`。
