# 优化总结与注意事项

## 0. 文档用途
总结本轮四个优化点的实现内容、实际收益、运行代价和使用注意事项。

## 1. 四个优化点总结

### 1.1 共享栈
- 实现内容：`Fiber` 增加共享栈开关与线程局部共享栈池，切换时做快照保存/恢复。
- 直接收益：大量短协程场景内存占用下降。
- 运行代价：切换增加栈内存拷贝开销。

### 1.2 协程嵌套
- 实现内容：新增 `call/back/parent`，在 `yield()` 中优先回父协程。
- 直接收益：父子协程可同步编排执行流程。
- 运行代价：状态机路径更复杂，需要严格线程内调用。

### 1.3 复杂调度算法
- 实现内容：增加 `FIFO/PRIORITY/MLFQ/EDF/HYBRID` 五种策略与 `ScheduleOptions`。
- 直接收益：可按业务特征选择任务执行顺序。
- 运行代价：调度决策复杂度高于单 FIFO 队列。

### 1.4 协程池
- 实现内容：新增 `CoroutinePool`，按参数 key 分桶缓存 `TERM` Fiber。
- 直接收益：回调任务场景减少频繁创建销毁。
- 运行代价：需要维护缓存上限和回收一致性。

## 2. 使用注意事项

### 2.1 共享栈
- 共享栈协程绑定首次运行线程，跨线程执行会触发断言。
- 栈对象在切出后依赖快照恢复，不应持久化栈地址。

### 2.2 协程嵌套
- `call()` 要求子协程处于 `READY`。
- 父子协程同时启用共享栈的嵌套路径会返回 `CALL_ERR_SHARED_NESTED_UNSUPPORTED`。

### 2.3 调度策略
- `scheduleLock()` 仍可用，等价于默认 `ScheduleOptions` 入队。
- MLFQ 参数会归一化：`levels` 与 `aging_sequence_gap` 最小值为 1。

### 2.4 协程池
- `release()` 仅回收 `TERM` 协程。
- 缓存 key 由 `stacksize + run_in_scheduler + use_shared_stack` 组成，参数要与 `acquire()` 保持一致。

## 3. 当前实现边界
- 共享栈+嵌套协程的双共享栈组合未开放。
- MLFQ 时间片仍采用“yield 后降级”模型，未引入真实时间片统计。
- 调度策略选择在全局任务向量上完成，任务量极大时选取成本会上升。

## 4. 升级结果结论
四项优化已全部接入主代码路径，并由独立测试覆盖关键行为：
- 共享栈：复用与快照一致性通过。
- 嵌套协程：父子与多层返回链路通过。
- 复杂调度：各策略顺序与配置归一化通过。
- 协程池：复用、上限、分桶隔离通过。
