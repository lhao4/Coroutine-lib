# 再开发问题复盘报告

## 1. 文档用途
统计本项目本轮再开发过程中的关键难点、问题点、思考过程和最终解决方案，并给出对应代码片段或代码位置。

  - `498e571`：共享栈 / 协程嵌套 / 复杂调度 / 协程池
  - `788f820`：`Fiber::call()` 从断言保护改为错误码语义

## 3. 问题总览
1. 共享栈快照需要精确获取 SP，且要兼容多架构寄存器布局。
2. 多协程复用同一共享栈槽位时，owner 切换容易导致栈数据污染。
3. 共享栈协程跨线程执行会破坏上下文与槽位归属。
4. 共享栈 + 协程复用（`reset`）路径下，context 重新初始化时机容易出错。
5. 协程嵌套下 `yield` 返回路径必须优先回父协程，不然会回错到调度协程。
6. “父子协程都用共享栈”属于未完成语义，最初仅靠 `assert`，Release 下会失效。
7. 调度器从 FIFO 扩展到多策略时，需要保持旧接口行为不变。
8. MLFQ 参数存在非法输入，需要归一化，否则会产生除零/无效层级问题。
9. 协程池复用需要解决 key 设计和复用污染问题。
10. 测试覆盖最初不足，新增能力需要可回归的自动化测试矩阵。
11. 文档存在“历史状态残留”，与当前实现不一致，需要系统性修正。

## 4. 关键问题与解决过程

### 4.1 共享栈 SP 获取与快照边界

#### 4.1.1 问题
共享栈切换要保存“活动栈区间”，前提是拿到正确 SP。不同 CPU 架构在 `ucontext_t` 中 SP 字段不同，直接写死会导致数据错误或崩溃。

#### 4.1.2 思考
- 必须用条件编译按架构读取 SP。
- 不支持的架构直接编译时报错，避免静默错误。

#### 4.1.3 解决
关键实现如下：

```cpp
static uintptr_t GetContextStackPointer(const ucontext_t& ctx) {
#if defined(__x86_64__) && defined(REG_RSP)
    return static_cast<uintptr_t>(ctx.uc_mcontext.gregs[REG_RSP]);
#elif defined(__i386__) && defined(REG_ESP)
    return static_cast<uintptr_t>(ctx.uc_mcontext.gregs[REG_ESP]);
#elif defined(__aarch64__)
    return static_cast<uintptr_t>(ctx.uc_mcontext.sp);
#else
#error "shared stack currently supports x86_64/i386/aarch64"
#endif
}
```

代码位置：`src/fiber.cpp:64-74`

### 4.2 共享栈 owner 切换与数据污染

#### 4.2.1 问题
多个共享栈协程竞争同一槽位时，如果新 owner 直接覆盖栈，会把旧 owner 的活动栈帧破坏。

#### 4.2.2 思考
owner 切换必须分两步：
- 旧 owner 先保存快照。
- 新 owner 再恢复自己的快照并接管 owner。

#### 4.2.3 解决
在 `prepareSharedStack()` 中显式处理 owner 切换：

```cpp
std::shared_ptr<Fiber> owner = slot->owner.lock();
if (owner && owner.get() != this) {
    if (owner.get() == t_fiber) {
        owner->saveCurrentSharedStackSnapshot();
    } else {
        owner->saveSharedStackSnapshot();
    }
}

if (!owner || owner.get() != this) {
    restoreSharedStackSnapshot();
    slot->owner = shared_from_this();
}
```

代码位置：`src/fiber.cpp:300-312`

### 4.3 共享栈跨线程执行约束

#### 4.3.1 问题
共享栈槽位是线程局部池管理，协程如果跨线程执行会破坏 owner 约束与 context 假设。

#### 4.3.2 思考
共享栈协程需要“首次运行线程绑定”，后续必须在同线程执行。

#### 4.3.3 解决
在 `prepareSharedStack()` 增加线程绑定校验：

```cpp
const pid_t tid = Thread::GetThreadId();
if (m_ownerThread == -1) {
    m_ownerThread = tid;
} else {
    assert(m_ownerThread == tid);
}
```

代码位置：`src/fiber.cpp:273-279`

### 4.4 共享栈 + reset 复用路径

#### 4.4.1 问题
`Fiber` 被复用后，可能仍持有旧共享槽位或旧 context 状态。若 `reset` 不重建 context，会出现切换异常。

#### 4.4.2 思考
- 独立栈：直接重建 context。
- 共享栈：若已有槽位，绑定槽位后重建；若无槽位，延迟到 `prepareSharedStack()` 初始化。

#### 4.4.3 解决
`reset()` 分支处理如下：

```cpp
if (!m_useSharedStack) {
    initFiberContext();
    return;
}

SharedStackSlot* slot = ToSharedStackSlot(m_sharedStackSlot);
if (slot) {
    m_stack = slot->stack;
    m_stacksize = static_cast<uint32_t>(slot->size);
    initFiberContext();
} else {
    m_ctxInitialized = false;
}
```

代码位置：`src/fiber.cpp:409-422`

### 4.5 协程嵌套返回链路

#### 4.5.1 问题
嵌套场景中子协程 `yield()` 如果仍走“回调度协程/主协程”旧路径，会导致返回链路错误。

#### 4.5.2 思考
嵌套调用必须把“返回父协程”作为更高优先级分支。

#### 4.5.3 解决
`yield()` 优先检测父协程路径：

```cpp
if (m_returnToParent && m_parent != nullptr) {
    back();
    return;
}
```

代码位置：`src/fiber.cpp:523-527`

父子关系建立和清理在 `call()` 中完成，位置：`src/fiber.cpp:491-504`

### 4.6 assert 在 Release 失效

#### 4.6.1 问题
最初“父子协程都使用共享栈”的未支持场景仅有 `assert`。Release 编译会移除断言，可能落入未定义行为。

#### 4.6.2 思考
底层高性能库不应依赖异常，且 Release 需要可生效保护。

#### 4.6.3 解决
将 `call()` 改为错误码 API，保留运行时硬保护：

```cpp
if (m_state != READY) {
    return CALL_ERR_NOT_READY;
}
...
if (m_useSharedStack && parent->m_useSharedStack) {
    return CALL_ERR_SHARED_NESTED_UNSUPPORTED;
}
```

代码位置：`src/fiber.cpp:468-485`

错误码定义：

```cpp
enum CallCode {
    CALL_OK = 0,
    CALL_ERR_NOT_READY = 1,
    CALL_ERR_NO_CURRENT_FIBER = 2,
    CALL_ERR_SELF_CALL = 3,
    CALL_ERR_SHARED_NESTED_UNSUPPORTED = 4
};
```

代码位置：`include/mycoroutine/fiber.h:38-45`

### 4.7 调度策略扩展与兼容性

#### 4.7.1 问题
新增 PRIORITY/MLFQ/EDF/HYBRID 时，不能破坏既有 `scheduleLock()` 用法。

#### 4.7.2 思考
- 新增 `scheduleEx(options)` 作为增强入口。
- `scheduleLock()` 保持语义，内部转发。

#### 4.7.3 解决
`scheduleLock()` 转发到 `scheduleEx()`：

```cpp
template <class FiberOrCb>
void scheduleLock(FiberOrCb fc, int thread = -1) {
    ScheduleOptions options;
    options.thread = thread;
    scheduleEx(fc, options);
}
```

代码位置：`include/mycoroutine/scheduler.h:141-147`

策略统一收敛在 `pickNextTaskLocked()`：`src/scheduler.cpp:280-418`

### 4.8 MLFQ 非法配置与饥饿问题

#### 4.8.1 问题
`levels=0` 或 `aging_sequence_gap=0` 会导致无效队列/除零风险；同时低优任务可能饥饿。

#### 4.8.2 思考
- 配置入口做归一化，保证下游逻辑有合法值。
- 选取时加入 aging 提升，限制饥饿。

#### 4.8.3 解决
配置归一化：

```cpp
if (normalized.levels == 0) {
    normalized.levels = 1;
}
if (normalized.aging_sequence_gap == 0) {
    normalized.aging_sequence_gap = 1;
}
```

代码位置：`src/scheduler.cpp:13-25`

### 4.9 协程池 key 设计与复用污染

#### 4.9.1 问题
复用池如果不按关键参数分桶，会把不同运行模式协程混用（如 shared/non-shared）。

#### 4.9.2 思考
key 必须包含：
- 栈大小（默认值归一化）
- `run_in_scheduler`
- `use_shared_stack`

#### 4.9.3 解决
`makeKey()` 按位编码：

```cpp
const uint64_t normalized = static_cast<uint64_t>(normalizeStackSize(stacksize)) & ((1ull << 60) - 1);
const uint64_t run_flag = run_in_scheduler ? (1ull << 62) : 0;
const uint64_t shared_flag = use_shared_stack ? (1ull << 63) : 0;
return normalized | run_flag | shared_flag;
```

代码位置：`src/coroutine_pool.cpp:20-26`

并在 `release()` 严格要求 `TERM`：`src/coroutine_pool.cpp:70-73`
