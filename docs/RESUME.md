# RESUME

## 文档目的

本文档用于将项目成果高质量转化为简历条目与面试表达，突出系统编程与协程运行时设计能力。

---

## 1. 简历项目描述（可直接使用）

实现了一个小型 C++17 协程运行时库（约 2000 行核心逻辑），基于 `ucontext` 构建有栈协程与上下文切换机制，设计并实现多线程调度器、`epoll + eventfd` 事件循环、定时器与系统调用 Hook，将阻塞式 socket IO 转换为协程挂起/恢复流程，形成“任务调度 + IO 复用 + 超时控制”的完整运行时闭环。

---

## 2. 技术亮点（3~5 条）

1. 基于 `ucontext` 实现协程上下文切换与生命周期管理（`READY/RUNNING/TERM`）。
2. 设计 `Scheduler + IOManager` 双层调度模型，统一执行计算任务、IO 事件和定时回调。
3. 封装 `epoll + eventfd` 事件驱动机制，支持跨线程唤醒与事件回调协程化。
4. 实现 Hook `read/write/connect/accept/sleep` 等阻塞调用，支持 `EAGAIN/超时/重试` 协程化流程。
5. 构建 `FdManager` 管理 fd 元信息（socket/nonblock/timeout），保证 Hook 行为一致性。

---

## 3. 面试讲解稿（口语化 2~3 分钟）

如果面试官让我介绍这个项目，我会这么讲：

我做的是一个小型协程运行时，不是业务框架，重点是把运行时核心链路做完整。项目基于 `ucontext` 实现有栈协程，每个协程有独立栈，通过 `swapcontext` 做上下文切换。调度层是一个多线程 `Scheduler`，任务既可以是协程，也可以是普通回调，回调会被包装成协程执行。

在 IO 侧我实现了 `IOManager`，它继承了 `Scheduler` 和 `TimerManager`，底层用 `epoll` 等待事件、用 `eventfd` 做线程唤醒。Hook 部分通过 `dlsym(RTLD_NEXT)` 拿到原始系统调用，把 `read/write/connect/accept` 这些阻塞调用改造成“注册事件 -> 当前协程 yield -> 事件就绪后恢复 -> 重试调用”的流程。

这个项目的难点在于协程状态一致性、调度与 IO 的协作、以及超时取消语义。我通过统一的 `do_io` 模板把 EAGAIN、EINTR、ETIMEDOUT 处理逻辑收敛在一条路径里。最终效果是：代码规模不大，但协程运行时最核心的能力都可验证。

如果继续演进，我会优先做调度策略升级、栈池、可观测性和更系统的并发测试。

## 4. 面试追问准备（问题 + 回答思路）

## Q1：为什么选择 `ucontext`，不直接用 C++20 协程？

回答思路：
- 本项目目标是学习运行时与调度机制，因此选用 `ucontext` 便于显式控制上下文。
- `ucontext` 可以清楚展示协程栈、状态机和切换点。
- C++20 协程更现代，但其运行时语义和编译器生成代码更复杂，不利于教学型项目“从零实现”目标。

## Q2：协程栈如何管理？

回答思路：
- 当前每个协程独立分配栈（默认 128KB），在协程析构时释放。
- 优点是实现简单直观；缺点是内存占用高。
- 后续可做栈池和 guard page。

## Q3：调度器为什么这样设计？

回答思路：
- 采用“统一任务抽象 + 协程/回调双入口”，降低用户接入成本。
- `IOManager` 继承 `Scheduler`，避免 IO 事件走另一套执行模型。
- 当前是单队列扫描，易读但性能一般，可演进到 work-stealing。

## Q4：协程相比线程优势是什么？

回答思路：
- 协程切换在用户态完成，开销一般低于线程上下文切换。
- 协程在高并发 IO 场景下可显著减少线程数量和栈内存占用。
- 但协程不是并行执行单元，仍需线程池配合 CPU 并行。

## Q5：这个项目和成熟协程库相比差异是什么？

回答思路：
- 本项目定位是学习型运行时原型，强调透明性和可读性。
- 成熟库在性能、生态、跨平台、观测和容错上更完善。
- 本项目价值在于我能完整解释从协程对象到事件循环再到 Hook 的每一段实现。

## Q6：你在这个项目中做过哪些“工程化改进”？

回答思路：
- 规范化 CMake 工程结构（安装导出、Presets、测试入口）。
- 修复关键运行时 bug（事件返回语义、定时器比较器、空指针路径）。
- 完善技术文档，把架构、模块、流程和使用方法落地。

## 6. 回答策略建议

面试时建议采用“问题 -> 设计 -> 实现 -> 权衡 -> 优化”结构：

1. 先说明痛点（阻塞 IO + 线程成本）
2. 再讲设计（协程 + 调度 + 事件循环 + 超时）
3. 再讲关键实现（`swapcontext`、`do_io`、`triggerEvent`）
4. 再讲边界和不足（调度策略、栈池、可观测性）
5. 最后给出演进计划（短中长期）

这套结构能同时体现系统思维、工程能力和自我审视能力。
