# 面试手册

## 0. 文档用途
用于面试时系统介绍 `mycoroutine` 项目，并准备从基础知识到项目全链路的高频问题与标准回答。

## 1. 项目介绍模板（可直接口述）

### 1.1 30 秒版本
我做了一个小型 C++17 协程运行时库，底层基于手写汇编（x86_64/AArch64）实现有栈协程上下文切换，上层实现了多线程调度器、`epoll + eventfd` 事件循环、定时器和 Hook。当前版本支持共享栈、协程嵌套、复杂调度策略（FIFO/PRIORITY/MLFQ/EDF/HYBRID）、协程池和协程同步原语（CoMutex/WaitGroup/Channel），并有 8 个自动化测试通过 Debug/ASan/TSan 三模式验证。Release 模式下 Fiber 上下文切换仅 17 ns/op。

### 1.2 3 分钟版本
这个项目目标不是业务框架，而是把协程运行时的核心链路做完整：协程对象、调度、IO 事件、超时、恢复。

协程层使用 `Fiber`，状态是 `READY/RUNNING/TERM`，通过手写 x86_64/AArch64 汇编实现上下文切换，只保存 callee-saved 寄存器，不走系统调用，Release 下切换仅 17 ns。调度层是 `Scheduler`，支持协程任务和回调任务；回调任务会通过 `CoroutinePool` 复用 `Fiber`，降低频繁创建销毁成本。

在 IO 侧，`IOManager` 继承 `Scheduler` 和 `TimerManager`，底层用 `epoll` 等待事件、`eventfd` 做唤醒，Hook 把阻塞调用改成”注册事件 -> yield -> 事件就绪后恢复”。

本轮重点做了七项核心能力：
1. 汇编上下文切换：替代 `ucontext`，消除 `sigprocmask` 系统调用，性能提升 22x。
2. 共享栈：线程内槽位复用 + 栈快照保存恢复，减少内存占用。
3. 协程嵌套：支持 `call/back` 父子协程同步调用。
4. 复杂调度：支持优先级、截止期、MLFQ、混合策略。
5. 协程池：按参数 key 分桶复用 `TERM` 协程。
6. 协程同步原语：`CoMutex`、`WaitGroup`、`Channel<T>`，等待时挂起协程不阻塞工作线程。
7. 任务队列优化：`deque` + move 语义，单线程调度吞吐提升 158x。

对边界也做了发布态保护：`call()` 返回错误码；父子协程同时共享栈时返回 `CALL_ERR_SHARED_NESTED_UNSUPPORTED`，不会在 Release 静默掉保护。Sanitizer 兼容性也做了完整适配（ASan fiber switch hook、TSan fiber tracking）。

### 1.3 10 分钟深挖结构
按这个顺序讲最稳：
1. 问题背景：阻塞 IO + 线程成本高。
2. 总体架构：`Fiber/Scheduler/IOManager/Hook/Timer/Sync`。
3. 关键机制：汇编上下文切换、事件驱动、超时恢复。
4. 七项核心能力：汇编切换、共享栈、嵌套、调度策略、协程池、同步原语、队列优化。
5. 性能数据：Fiber 切换 17 ns/op，Scheduler 单线程 527 ns/op（Release）。
6. 质量保障：`ctest` 8 项回归，Debug/ASan/TSan 三模式全量通过。
7. 边界约束：共享栈嵌套组合未开放，返回错误码。

## 2. 从基础到项目的面试问题与回答

## 2.1 基础层（C++/系统）

### Q1：协程和线程的本质区别是什么？
A：线程是内核调度单位，协程是用户态调度单位。线程切换需要内核参与；协程在用户态切换，开销通常更低。协程本身不提供并行，CPU 并行仍依赖多线程。

### Q2：为什么做有栈协程而不是无栈协程？
A：有栈协程可以保持同步调用风格，业务代码迁移成本低，递归和深调用链更自然。代价是栈内存管理更复杂，所以后续引入了共享栈来降低内存成本。

### Q3：为什么用汇编替代 `ucontext`？
A：`ucontext` 的 `swapcontext` 内部会调用 `sigprocmask` 保存/恢复信号掩码，这是一个系统调用。协程切换不需要修改信号掩码，所以用汇编直接保存/恢复 callee-saved 寄存器和栈指针，避免内核态开销。实测 Release 下从 385 ns 降到 17 ns，提升约 22 倍。

### Q4：`thread_local` 在这个项目中解决了什么？
A：用于保存每线程独立运行态，如当前协程、主协程、调度协程、共享栈池。避免跨线程共享状态带来的竞争和错误。

### Q5：为什么这里大量使用 `shared_ptr`？
A：协程执行过程中存在多方持有（调度器、当前运行上下文、池复用等），`shared_ptr` 让生命周期管理更安全，尤其是切换期间防止悬空。

## 2.2 架构层

### Q6：项目核心模块有哪些，如何协作？
A：
- `Fiber`：协程执行与切换。
- `Scheduler`：任务队列、线程调度、策略选取。
- `IOManager`：`epoll` 事件与定时器驱动恢复。
- `Hook`：把阻塞调用转成协程挂起恢复。
- `CoroutinePool`：复用回调协程。
流程是：任务入队 -> 调度选取 -> 协程执行 -> IO 挂起 -> 事件就绪恢复。

### Q7：`Scheduler` 为什么要支持回调和协程两种任务？
A：为了兼容现有调用方式。上层可以直接丢回调，不需要先手动包装协程。底层统一执行模型，回调路径最终也会进入协程执行。

### Q8：调度策略是怎么扩展进去的？
A：新增 `SchedulePolicy` 与 `ScheduleOptions`，在 `pickNextTaskLocked()` 内统一策略比较；保留 `scheduleLock()`，内部转发到 `scheduleEx()`，确保兼容性。

## 2.3 机制层（重点）

### Q9：共享栈是怎么实现的？
A：每线程维护共享栈槽位池。协程切入前如果槽位 owner 变化，会先保存旧 owner 活动栈快照，再恢复当前协程快照，最后切换执行。共享栈快照在 ASan 下需要先调用 `__asan_unpoison_memory_region` 避免堆分配栈中的红区标记干扰 memcpy。

位置：`src/fiber.cpp` 的 `prepareSharedStack()`。

### Q10：共享栈怎么避免跨线程错误？
A：首次运行时记录 `m_ownerThread`，后续运行线程必须一致，否则断言失败。共享栈池本身也是线程局部对象。

### Q11：协程嵌套怎么保证返回父协程？
A：`call()` 建立父子关系并切入；`yield()` 优先检测 `m_returnToParent && m_parent`，满足则走 `back()` 回父协程。

### Q12：为什么 `call()` 改成错误码？
A：因为底层高性能场景下不能依赖异常，也不能只靠 `assert`。`assert` 在 Release 会被裁掉，错误码能保证发布态也有明确保护语义。

### Q13：`call()` 有哪些错误码？
A：
- `CALL_OK`
- `CALL_ERR_NOT_READY`
- `CALL_ERR_NO_CURRENT_FIBER`
- `CALL_ERR_SELF_CALL`
- `CALL_ERR_SHARED_NESTED_UNSUPPORTED`

定义位置：`include/mycoroutine/fiber.h`。

### Q14：为什么“父子都共享栈”现在不支持？
A：这个组合的快照时机和 parent/child 切换顺序耦合更深，当前实现先给出确定失败语义（错误码），避免未定义行为。属于清晰边界，而不是隐式 bug。

### Q15：MLFQ 里怎么避免低优任务饿死？
A：引入 aging，按任务等待序列差提升有效层级。并且 `levels`、`aging_sequence_gap` 在配置入口做归一化，防止非法配置。

### Q16：协程池怎么避免复用污染？
A：按 `(stacksize, run_in_scheduler, use_shared_stack)` 做 key 分桶，只有同配置对象才会复用；并且 `release()` 只回收 `TERM` 协程。

## 2.4 测试与工程化层

### Q17：你怎么证明这些优化是可用的？
A：共 8 个自动化测试全部通过三种模式验证：
- `smoke` / `shared_stack` / `nested` / `policy` / `pool` / `sync` / `hook` / `stress`
- Debug：8/8 通过
- ASan：7/8 通过（1 项 LSan 保守扫描误报，已验证为 false positive）
- TSan：8/8 通过，无数据竞争
- 还有 Benchmark 提供量化性能数据：Fiber 切换 17 ns/op，Scheduler 单线程 527 ns/op。

### Q18：嵌套失败路径有测吗？
A：有。`tests/nested/main.cpp` 覆盖了父子都共享栈时 `call()` 返回 `CALL_ERR_SHARED_NESTED_UNSUPPORTED`。

### Q19：项目现在的边界有哪些？
A：
- 父子协程同时共享栈的嵌套调用未开放。
- 调度任务容器为 `deque` + 全局锁扫描模型，超大任务量下锁竞争是瓶颈。
- 运行时指标导出仍较少，主要靠日志。
- 仅支持 Linux + x86_64/AArch64。

## 2.5 追问与反问（高频）

### Q20：如果面试官问“这不是重复造轮子吗？”
A：这是运行时实现项目，目标是证明我具备从协程切换、调度、IO 到错误边界的系统实现能力，不是替代成熟库。价值在可解释性和工程落地能力。

### Q21：你在这个项目里最关键的工程决策是什么？
A：把“未支持语义”从 Debug 断言提升为 Release 可生效的错误码契约。这保证了线上行为确定性，是底层库最关键的可靠性原则之一。

### Q22：如果问”你最难的 bug 是什么？”
A：汇编上下文切换后的 ASan 兼容性问题。替换 `ucontext` 后 ASan 报 “starting fiber switch while in fiber switch”，根本原因是新协程首次执行时直接从 `FiberEntryPoint` 开始，不会经过 `swapWithSanitizer` 中的 `finish_switch_fiber` 调用。需要理解 ASan fiber switch hook 的状态机才能定位，最终在 `MainFunc` 入口补充了 `finish_switch_fiber` 调用。另一个难点是共享栈快照的 ASan 误报——堆分配的栈中 ASan 植入的红区标记会干扰 memcpy，需要用 `__asan_unpoison_memory_region` 解决。

## 6. 回答策略

面试时建议采用"问题 -> 设计 -> 实现 -> 权衡 -> 优化"结构：

1. 先说明痛点（阻塞 IO + 线程成本）
2. 再讲设计（协程 + 调度 + 事件循环 + 超时 + 同步原语）
3. 再讲关键实现（汇编 context swap、`do_io`、`triggerEvent`、`CoMutex`/`Channel`）
4. 用数据说话（17 ns 切换、527 ns 调度、8 项测试 × 3 模式）
5. 再讲边界和不足（无锁队列、栈池、io_uring）
6. 最后给出演进计划（短中长期）

这套结构能同时体现系统思维、工程能力和自我审视能力。
