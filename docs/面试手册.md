# 面试手册

## 0. 文档用途
用于面试时系统介绍 `mycoroutine` 项目，并准备从基础知识到项目全链路的高频问题与标准回答。

## 1. 项目介绍模板（可直接口述）

### 1.1 30 秒版本
我做了一个小型 C++17 协程运行时库，底层基于 `ucontext` 实现有栈协程和上下文切换，上层实现了多线程调度器、`epoll + eventfd` 事件循环、定时器和 Hook。当前版本新增了共享栈、协程嵌套、复杂调度策略（FIFO/PRIORITY/MLFQ/EDF/HYBRID）和协程池，并有 6 个自动化测试覆盖核心能力。

### 1.2 3 分钟版本
这个项目目标不是业务框架，而是把协程运行时的核心链路做完整：协程对象、调度、IO 事件、超时、恢复。

协程层使用 `Fiber`，状态是 `READY/RUNNING/TERM`，通过 `swapcontext` 切换。调度层是 `Scheduler`，支持协程任务和回调任务；回调任务会通过 `CoroutinePool` 复用 `Fiber`，降低频繁创建销毁成本。

在 IO 侧，`IOManager` 继承 `Scheduler` 和 `TimerManager`，底层用 `epoll` 等待事件、`eventfd` 做唤醒，Hook 把阻塞调用改成“注册事件 -> yield -> 事件就绪后恢复”。

本轮重点优化了四点：
1. 共享栈：线程内槽位复用 + 栈快照保存恢复，减少内存占用。
2. 协程嵌套：支持 `call/back` 父子协程同步调用。
3. 复杂调度：支持优先级、截止期、MLFQ、混合策略。
4. 协程池：按参数 key 分桶复用 `TERM` 协程。

对边界也做了发布态保护：`call()` 返回错误码；父子协程同时共享栈时返回 `CALL_ERR_SHARED_NESTED_UNSUPPORTED`，不会在 Release 静默掉保护。

### 1.3 10 分钟深挖结构
按这个顺序讲最稳：
1. 问题背景：阻塞 IO + 线程成本高。
2. 总体架构：`Fiber/Scheduler/IOManager/Hook/Timer`。
3. 关键机制：上下文切换、事件驱动、超时恢复。
4. 四项优化：共享栈、嵌套、调度策略、协程池。
5. 质量保障：`ctest` 6 项回归，覆盖成功与失败路径。
6. 边界约束：共享栈嵌套组合未开放，返回错误码。

## 2. 从基础到项目的面试问题与回答

## 2.1 基础层（C++/系统）

### Q1：协程和线程的本质区别是什么？
A：线程是内核调度单位，协程是用户态调度单位。线程切换需要内核参与；协程在用户态切换，开销通常更低。协程本身不提供并行，CPU 并行仍依赖多线程。

### Q2：为什么做有栈协程而不是无栈协程？
A：有栈协程可以保持同步调用风格，业务代码迁移成本低，递归和深调用链更自然。代价是栈内存管理更复杂，所以后续引入了共享栈来降低内存成本。

### Q3：`ucontext` 做了什么？
A：`ucontext` 保存和恢复执行上下文（寄存器、栈指针、程序计数器），通过 `swapcontext` 在协程之间切换。

### Q4：`thread_local` 在这个项目中解决了什么？
A：用于保存每线程独立运行态，如当前协程、主协程、调度协程、共享栈池。避免跨线程共享状态带来的竞争和错误。

### Q5：为什么这里大量使用 `shared_ptr`？
A：协程执行过程中存在多方持有（调度器、当前运行上下文、池复用等），`shared_ptr` 让生命周期管理更安全，尤其是切换期间防止悬空。

## 2.2 架构层

### Q6：项目核心模块有哪些，如何协作？
A：
- `Fiber`：协程执行与切换。
- `Scheduler`：任务队列、线程调度、策略选取。
- `IOManager`：`epoll` 事件与定时器驱动恢复。
- `Hook`：把阻塞调用转成协程挂起恢复。
- `CoroutinePool`：复用回调协程。
流程是：任务入队 -> 调度选取 -> 协程执行 -> IO 挂起 -> 事件就绪恢复。

### Q7：`Scheduler` 为什么要支持回调和协程两种任务？
A：为了兼容现有调用方式。上层可以直接丢回调，不需要先手动包装协程。底层统一执行模型，回调路径最终也会进入协程执行。

### Q8：调度策略是怎么扩展进去的？
A：新增 `SchedulePolicy` 与 `ScheduleOptions`，在 `pickNextTaskLocked()` 内统一策略比较；保留 `scheduleLock()`，内部转发到 `scheduleEx()`，确保兼容性。

## 2.3 机制层（重点）

### Q9：共享栈是怎么实现的？
A：每线程维护共享栈槽位池。协程切入前如果槽位 owner 变化，会先保存旧 owner 活动栈快照，再恢复当前协程快照，最后切换执行。

关键代码片段：

```cpp
std::shared_ptr<Fiber> owner = slot->owner.lock();
if (owner && owner.get() != this) {
    if (owner.get() == t_fiber) {
        owner->saveCurrentSharedStackSnapshot();
    } else {
        owner->saveSharedStackSnapshot();
    }
}
if (!owner || owner.get() != this) {
    restoreSharedStackSnapshot();
    slot->owner = shared_from_this();
}
```

位置：`src/fiber.cpp` 的 `prepareSharedStack()`。

### Q10：共享栈怎么避免跨线程错误？
A：首次运行时记录 `m_ownerThread`，后续运行线程必须一致，否则断言失败。共享栈池本身也是线程局部对象。

### Q11：协程嵌套怎么保证返回父协程？
A：`call()` 建立父子关系并切入；`yield()` 优先检测 `m_returnToParent && m_parent`，满足则走 `back()` 回父协程。

### Q12：为什么 `call()` 改成错误码？
A：因为底层高性能场景下不能依赖异常，也不能只靠 `assert`。`assert` 在 Release 会被裁掉，错误码能保证发布态也有明确保护语义。

### Q13：`call()` 有哪些错误码？
A：
- `CALL_OK`
- `CALL_ERR_NOT_READY`
- `CALL_ERR_NO_CURRENT_FIBER`
- `CALL_ERR_SELF_CALL`
- `CALL_ERR_SHARED_NESTED_UNSUPPORTED`

定义位置：`include/mycoroutine/fiber.h`。

### Q14：为什么“父子都共享栈”现在不支持？
A：这个组合的快照时机和 parent/child 切换顺序耦合更深，当前实现先给出确定失败语义（错误码），避免未定义行为。属于清晰边界，而不是隐式 bug。

### Q15：MLFQ 里怎么避免低优任务饿死？
A：引入 aging，按任务等待序列差提升有效层级。并且 `levels`、`aging_sequence_gap` 在配置入口做归一化，防止非法配置。

### Q16：协程池怎么避免复用污染？
A：按 `(stacksize, run_in_scheduler, use_shared_stack)` 做 key 分桶，只有同配置对象才会复用；并且 `release()` 只回收 `TERM` 协程。

## 2.4 测试与工程化层

### Q17：你怎么证明这些优化是可用的？
A：新增了 5 个专项测试并纳入 `ctest`：
- `shared_stack`
- `nested`
- `policy`
- `pool`
- `hook`（覆盖 Hook/IOManager/timer/fd_manager 场景）
加上 `smoke` 共 6 项，当前 `6/6 passed`。

### Q18：嵌套失败路径有测吗？
A：有。`tests/nested/main.cpp` 覆盖了父子都共享栈时 `call()` 返回 `CALL_ERR_SHARED_NESTED_UNSUPPORTED`。

### Q19：项目现在的边界有哪些？
A：
- 父子协程同时共享栈的嵌套调用未开放。
- 调度任务容器仍是向量扫描模型，超大任务量下选取成本较高。
- 运行时指标导出仍较少，主要靠日志。

## 2.5 追问与反问（高频）

### Q20：如果面试官问“这不是重复造轮子吗？”
A：这是运行时实现项目，目标是证明我具备从协程切换、调度、IO 到错误边界的系统实现能力，不是替代成熟库。价值在可解释性和工程落地能力。

### Q21：你在这个项目里最关键的工程决策是什么？
A：把“未支持语义”从 Debug 断言提升为 Release 可生效的错误码契约。这保证了线上行为确定性，是底层库最关键的可靠性原则之一。

### Q22：如果问“你最难的 bug 是什么？”
A：共享栈 owner 切换导致的栈数据污染风险。核心是切换前后快照时机必须严格，最终在 `prepareSharedStack()` 中明确了 owner 切换流程并通过专项测试验证。

## 6. 回答策略

面试时建议采用"问题 -> 设计 -> 实现 -> 权衡 -> 优化"结构：

1. 先说明痛点（阻塞 IO + 线程成本）
2. 再讲设计（协程 + 调度 + 事件循环 + 超时）
3. 再讲关键实现（`swapcontext`、`do_io`、`triggerEvent`）
4. 再讲边界和不足（调度策略、栈池、可观测性）
5. 最后给出演进计划（短中长期）

这套结构能同时体现系统思维、工程能力和自我审视能力。
