# KNOWN_LIMITATIONS

## 文档目的

本文档汇总 `mycoroutine` 在当前版本（计划发布 `v1.0.0`）下的已知不足与边界，作为发布说明和后续迭代输入。

---

## 1. v1.0.0 版本定位

- 类型：学习与工程实践导向的小型协程运行时
- 目标：打通协程对象、调度、IO 事件、定时器、Hook 的完整链路
- 非目标：直接作为生产环境高可靠基础库

---

## 2. 当前已知不足

## 2.1 自动化测试覆盖不足

- 当前 `ctest` 主要覆盖 `mycoroutine_smoke_test`。
- benchmark 与 HTTP demo 仍以手工测试流程为主，未纳入统一 CI 回归矩阵。

影响：
- 回归保护能力有限，复杂场景下问题可能滞后暴露。

## 2.2 性能结果稳定性不足

- 已完成性能测试并得到有效数据，但共享环境下波动明显（尤其是调度回调吞吐）。
- 尚未建立固定基准机、CPU 绑定和多轮统计的标准化流程。

影响：
- 当前数据适合“同环境相对对比”，不适合直接作为绝对性能承诺。

## 2.3 调度器扩展性一般

- 调度任务容器仍以全局队列 + 线性扫描为主。
- 多 worker 下扩展性受锁竞争和队列结构影响，未形成线性提升。

影响：
- 高并发下吞吐上限和公平性表现受限。

## 2.4 协程栈管理较基础

- 协程栈采用固定分配与释放，缺少栈池复用与 guard page。
- 大量短生命周期协程场景可能产生分配抖动。

影响：
- 内存效率与极端场景鲁棒性仍有提升空间。

## 2.5 Hook 兼容性边界未系统化覆盖

- 已支持核心阻塞调用协程化，但与更多第三方库组合的兼容矩阵尚不完整。
- 线程局部 Hook 开关特性需要调用方严格遵守执行上下文。

影响：
- 复杂集成场景可能出现行为差异，需要额外验证。

## 2.6 可观测性能力较弱

- 当前以日志为主，缺少系统化指标、trace、profiling 入口。
- 对队列长度、协程切换次数、慢任务分布等关键运行态数据暴露不足。

影响：
- 问题定位成本较高，不利于持续性能优化。

## 2.7 错误处理风格不统一

- 代码中同时存在 `assert`、返回值、日志等多种错误处理路径。
- 一些路径更偏向“开发期快速失败”，对线上容错不友好。

影响：
- 异常场景可控性和一致性不足。

## 2.8 平台依赖明显

- 当前实现依赖 Linux 机制（`ucontext`、`epoll`、`eventfd`、`dlsym`）。
- 非 Linux 平台未做兼容性设计。

影响：
- 跨平台能力有限。

---

## 3. 发布结论（v1.0.0）

可发布为 `v1.0.0`，理由如下：

1. 核心功能链路完整且可运行：协程、调度、IO、定时器、Hook 已贯通。
2. 系统测试与性能测试已完成，并已形成测试文档。
3. 当前不足均已显式记录，项目边界清晰，不存在“能力误导”。

发布声明建议：
- `v1.0.0` 为“功能闭环版本（baseline）”，后续将以稳定性、测试覆盖、可观测性和调度扩展性为主要演进方向。

---

## 4. v1.1 优先级建议

1. 将关键 benchmark 与网络连通测试纳入自动化流程（含多轮统计）。
2. 补齐 Hook/定时器/事件取消路径的单元与回归测试。
3. 增强调度器队列结构与跨线程唤醒策略，降低锁竞争。
4. 引入基础运行指标（切换次数、队列长度、慢任务分位）。
