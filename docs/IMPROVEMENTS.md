# IMPROVEMENTS

## 文档目的

本文档从系统编程角度评估当前实现的优势、不足和演进方向，并给出短中长期优化建议与优先级。

---

## 1. 当前实现的优点

1. 运行时链路完整：协程、调度、IO、超时、Hook 都已贯通。
2. 代码路径直接：关键函数集中，便于阅读和调试。
3. 事件驱动明确：`epoll + eventfd` 组合清晰。
4. 可扩展性基础已具备：`IOManager` 继承 `Scheduler + TimerManager`。
5. 实际可运行：有 smoke test 和示例程序可验证。

---

## 2. 当前实现的不足

1. 调度策略较基础：全局任务容器 + 线性扫描。
2. 栈管理简单：协程栈固定分配，缺少池化与保护页。
3. 定时器时钟源为 `system_clock`，受系统时间调整影响。
4. 错误处理风格不统一（`assert`、返回值、日志混用）。
5. Hook 兼容性边界未系统化验证（与第三方库组合场景）。
6. 可观测性较弱：缺少结构化指标、trace、profiling 钩子。
7. 测试覆盖偏轻：主要是 smoke，缺少并发/压力/回归测试。

---

## 3. 设计取舍分析

## 3.1 为什么当前实现“简单但可用”

- 用 `ucontext` 换取协程控制流可解释性
- 用 `vector` 任务队列换取实现复杂度低
- 用 `assert` 快速暴露不变量错误

## 3.2 代价

- 性能和公平性不是最优
- 生产环境容错能力不足
- 可移植性和可观测性欠缺

---

## 4. 优化方向（按主题）

## 4.1 调度策略

- 从单全局队列改为“每线程本地队列 + work-stealing”
- 引入优先级或时间片策略
- 增加任务饥饿检测

## 4.2 协程栈管理

- 栈池复用减少 `malloc/free` 抖动
- 按任务类型配置栈大小
- 引入 guard page 检测栈溢出

## 4.3 内存管理

- 明确 `IOManager::FdContext` 生命周期所有权（可转 `unique_ptr`）
- 减少热点路径临时对象分配

## 4.4 上下文切换性能

- 评估 `ucontext` 开销并建立基准
- 可选引入手写汇编上下文切换（平台相关）

## 4.5 多线程调度支持

- 改善跨线程任务唤醒策略
- 降低全局锁竞争
- 补充线程亲和任务分配策略

## 4.6 I/O 协程化能力

- 扩展 Hook 覆盖范围与一致性测试
- 明确 Hook 线程上下文约束的开发者提示

## 4.7 定时器/事件循环集成

- 使用 `steady_clock` 替换 `system_clock`
- 定时器容器可评估更高效结构（小根堆等）

## 4.8 可观测性

- 增加调度器指标（队列长度、活跃协程数、切换次数）
- 增加慢任务检测日志
- 预留 trace/span 接口

## 4.9 API 易用性

- 区分“库内部接口”和“用户稳定接口”
- 补充返回值语义和错误码约定

## 4.10 可测试性

- 增加单元测试：状态机、超时、取消路径
- 增加并发回归测试和压力测试
- 在 CI 加入 ASan/TSan/UBSan

## 4.11 可扩展性

- 提供可插拔调度策略接口
- 提供可插拔日志后端
- 将 Hook 策略配置化

---

## 5. 优化优先级建议

## 短期（1~2 周，优先做）

1. 定时器时钟改 `steady_clock`
2. 统一错误处理语义（减少裸 `assert`）
3. 补齐核心路径测试（调度/超时/取消）
4. 增加关键运行指标日志

## 中期（2~6 周）

1. 调度队列从全局线性扫描演进为分片或本地队列
2. 协程栈池与栈大小策略
3. Hook 兼容性和回归测试矩阵

## 长期（6 周以上）

1. 可插拔调度器策略框架
2. 更高性能上下文切换实现（可选）
3. 运行时诊断平台化（trace + profile + metrics）

---

## 6. 建议执行顺序（工程落地）

1. 先补可观测性和测试，建立“改动可验证”基础
2. 再改调度策略和栈管理，避免盲目性能优化
3. 最后做接口层和扩展层改造，防止中途 API 反复变动

---

## 7. 风险提示

- 调度策略和上下文切换改造风险最高，容易引入隐蔽并发 bug。
- Hook 扩展需谨慎处理与系统库、第三方库的兼容性。
- 性能优化必须依赖可重复基准，否则容易“优化错方向”。
